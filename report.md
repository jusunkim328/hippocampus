Gemini와 Codex가 독립적으로 동일한 문제를 지적:
- 15번과 8번이 "주관적 텍스트 → 시맨틱 매칭"이라는 동일 장르로 겹침
- 기술 실행력(30%)에서 ES|QL/멀티스텝을 깊이 보여줄 아이디어가 부족

**합의 진전**: 15번 만장일치 확정, 8번 탈락 합의, 21번 유력 2위. 3위 미합의.

### 라운드 2

Claude가 8번 탈락 수용, 21번 수용. 3위는 1번 유지하며 "감성(15) + 분석(21) + 발견(1)" 포트폴리오 논리 제시. 23번은 21번과 "분석적" 성격이 겹치고, arXiv 데이터 확보 용이성에서 1번이 우위라고 주장.

Gemini가 기존 23번 주장을 철회하고 1번에 동의. "검색 엔진 해커톤에서 '발견'을 재정의"라는 메타 내러티브가 강력하다는 데 공감. RRF 하이브리드 전략 + 참신성 검증(Novelty Validation) 단계를 보완으로 제안.

Codex도 조건부 동의. 단, "브리지 경로 강제 제시 + 단계별 근거 로그"가 필수이며, 데모에서 랜덤하게 보이면 플랜B(10 또는 23)로 즉시 스왑할 수 있어야 한다고 권고.

**합의 진전**: 15+21+1 **3자 수렴 완료**.

## 라운드별 전문

### 라운드 1

**Claude:**

23개를 평가할 때 5개 축으로 점수를 매겼다.
1. ES 기술 활용 깊이: 시맨틱 검색만 쓰면 낮음, ES|QL + Workflows + 어그리게이션까지 쓰면 높음
2. 데모 임팩트 (3분): 3분 안에 "와" 할 수 있는가
3. 혁신성/의외성: 심사위원이 "이건 처음 보네" 할 만한가
4. 데이터 확보 용이성: 14일 내 오픈 데이터로 구현 가능한가
5. 소셜 공유 가능성: 사람들이 결과를 공유하고 싶어하는가

탈락 그룹: 2번(민감 주제), 4번(문헌 정제 비용), 7번(인과 데이터 난이도), 10~14번(구체성 부족), 18번(14번 중복), 19번(11번 중복), 22번(5번 유사).

후보 그룹 8개를 스코어링한 결과:

| # | 이름 | 기술 | 데모 | 혁신 | 데이터 | 소셜 | 합계 |
|---|------|-----|------|------|--------|------|------|
| 1 | Serendipity Engine | 4 | 4 | 4 | 5 | 3 | 20 |
| 3 | Time Warp News | 3 | 5 | 3 | 4 | 4 | 19 |
| 5 | Deja Vu Detector | 3 | 4 | 3 | 3 | 3 | 16 |

최종 선택: 15(22점) + 8(21점) + 1(20점). 단, 15와 8의 감성 매칭 중복을 자인.

**Gemini:**

Claude의 선택은 '스토리텔링형 아이디어'에 집중되어 있어 대중성과 데모 직관성은 탁월하나, 기술 실행력(30%) 지표에서 안전한 선택에 머물렀다는 인상. 15번과 8번은 본질적으로 '도메인 간 벡터 매칭'이라는 유사한 기술 구조를 공유하여 다양성 부족.

15번 1순위 동의 — 벡터 검색(kNN)을 가장 섹시하게 풀어낼 수 있는 컨셉.

21번(Unseen Gap Detector) 강력 추천 — ES|QL의 강력한 집계 기능과 Agent Builder의 추론 능력을 증명할 최고의 기회. ES|QL로 밀도가 낮은 좌표를 추출하고 에이전트가 멀티스텝으로 분석하는 과정이 기술 실행력을 극대화.

23번(Conflict Catalyst) 제안 — 1번보다 구체적이고 실용적. 서로 다른 두 트렌드를 ES|QL로 기간별 집계하고 충돌/시너지 지점을 추론. 리스크 관리라는 실제 문제 해결로 영향력 점수 기대.

대안 조합: 15(대중성) + 21(기술) + 23(영향력).

**Codex:**

Claude의 프레임워크는 해커톤 심사 비중과 잘 맞으나, 15와 8이 "같은 장르"로 보일 위험. 15의 데이터 라이선스/정제 난이도와 크로스 도메인 정렬 문제가 생각보다 큼.

15 동의. 단, 데이터 라이선스가 명확한 2개 도메인만으로 범위 축소 필수. 크로스 도메인 정렬은 LLM으로 "중간 표현(설명 문장) 생성 → ELSER/벡터로 검색"하는 파이프라인으로 우회 가능.

1번은 2-hop 검색 구조 + 연결 그래프 시각화 필수. "우연=랜덤" 리스크 방지.

"감성 바이럴 1개 + 기술/분석 1개 + 발견/연결 1개" 포트폴리오 구성 제안. 15 또는 8 중 하나만 남기고, 나머지를 10(Sound of Silence) 또는 21(Unseen Gap Detector)로 대체.

### 라운드 2

**Claude:**

Gemini/Codex 피드백 반영하여 8번 탈락, 21번 수용.

3위 쟁점 분석:
- 1번: arXiv 데이터 용이, 2-hop 멀티스텝 자연스러움, "검색 패러다임 전복" 메타 내러티브
- 23번: ES|QL JOIN 자연스러움, 15/21과 기술 구조 다름. 단 트렌드 데이터 확보 난이도 높음
- 10번: 시계열 ES|QL로 깔끔. 단 소셜 공유 낮고 대시보드 느낌 위험

1번 유지 이유 세 가지:
1. "검색 엔진 회사 해커톤에서 검색을 뒤집는다"는 메타 내러티브가 킬러 스토리라인
2. 감성(15) + 분석(21) + 발견(1)로 세 축이 완벽 분리. 23번은 21번과 분석 성격 중복
3. arXiv 데이터 확보 + 2-hop 구조가 14일 스프린트에 현실적. 23번은 데이터 정제 병목 위험

최종: 15+21+1. ES 기능도 시맨틱 검색(15) + ES|QL 집계(21) + 하이브리드/RRF(1)로 고르게 커버.

**Gemini:**

15+21+1 조합을 해커톤 심사 기준의 기술성, 영향력, 데모 품질을 완벽하게 안배한 '전략적 포트폴리오'로 평가.

23번 기존 주장 철회 — 트렌드 데이터 노이즈가 심해 3분 데모에서 신뢰성 증명 어려움. 1번은 arXiv 같은 검증된 학술 데이터로 결과 품질 제어 용이.

1번 보완: RRF를 활용해 'BM25 저득점 + Vector 고득점' 항목(= 의외하지만 연결된 결과) 추출. 멀티스텝 에이전트가 연결고리 생성 후 ES|QL로 "이 두 개념이 동시 등장한 논문이 전무함"을 확인하는 **참신성 검증(Novelty Validation)** 단계 추가 제안.

구체화: 15번은 "무라카미 하루키 문체의 여행지 추천", 21번은 "AI 윤리의 빈틈", 1번은 "균류학 신경망과 분산 컴퓨팅의 유사성"처럼 직관적 예시 설정.

최종 스토리라인: "당신이 원하는 것을 찾고(15), 데이터 속에 숨겨진 구멍을 발견하며(21), 생각지 못한 연결을 제안한다(1)".

**Codex:**

15+21 확정 동의. 1번은 조건부 동의 — 리스크 관리 필수.

보완 요구사항:
1. 1번은 "유사도 추천"이 아니라 "브리지(연결 고리) 증명"에 초점. A–B 사이 2~3개 중간 노드를 반드시 제시하고, 각 hop마다 ES 결과를 화면에 남겨 '추론된 경로'처럼 보이게 할 것
2. 21번은 문제 정의를 끝까지 좁힐 것. "갭 → 검증 → 제안"이 한 화면에서 닫히는 시각화 준비
3. 15번은 2개 도메인 축소 필수. "정렬 실패"가 즉시 신뢰를 무너뜨림
4. 플랜B 준비 — 1번이 랜덤하게 보이면 10번 또는 23번으로 즉시 스왑

결론: 15+21+1은 우승권 조합이나, 1번은 "브리지 경로 강제 제시 + 단계별 근거 로그"가 없으면 15번의 그림자에 가려질 가능성 있음.

## 3자 합의 결론

### 최종 후보 3개

#### 1위: Taste Translator (취향 번역기) — #15
> 감성/소셜 축 | 데모 품질(30%) + 소셜 공유(10%) 공략

**3자 만장일치.** 한 도메인의 취향을 다른 도메인으로 번역하는 에이전트.

핵심 구현 전략:
- 도메인 2개로 축소 (라이선스 명확한 오픈 데이터만)
- LLM 중간 표현 생성으로 크로스 도메인 정렬 우회
- 멀티스텝: 취향 요약 → 중간 표현 생성 → 하이브리드 검색+RRF → 근거 스니펫 → 결과 카드
- ES|QL 집단 취향 집계로 기술적 무게감 추가

#### 2위: Unseen Gap Detector (데이터 진공 검색) — #21
> 기술/분석 축 | 기술 실행력(30%) 공략

**3자 만장일치.** 벡터 공간의 빈 공간을 찾아 미개척 영역을 제안하는 에이전트.

핵심 구현 전략:
- 문제 정의 좁히기 ("특허/논문 중 미개척 연구 영역")
- 5단계 멀티스텝: 분포/밀도 계산 → 공백 후보 추출 → 주변 근거 수집 → 반증 검색 → 제안 생성
- ES|QL + 집계 극한 활용 (missing/terms/date_histogram)
- 2D 투영 시각화로 "갭 → 검증 → 제안" 한 화면 완결

#### 3위: Serendipity Engine (우연한 발견 엔진) — #1
> 발견/혁신 축 | 영향력·혁신성(30%) 공략

**Claude+Gemini 합의, Codex 조건부 동의.**

핵심 구현 전략:
- 2-hop 검색: 첫 검색 결과에서 핵심 개념 추출 → 다른 코퍼스에서 재검색
- RRF 하이브리드: BM25 저득점 + Vector 고득점 = "의외하지만 연결된" 결과
- 참신성 검증: ES|QL로 "이 두 개념이 동시에 등장한 논문이 전무함" 확인
- 브리지 경로 강제 제시: A→B 사이 2~3개 중간 노드, hop마다 ES 결과 로그 표시
- 플랜B: 데모에서 랜덤하게 보이면 10번(Sound of Silence) 또는 23번(Conflict Catalyst)으로 스왑

### 포트폴리오 구조

| 축 | 아이디어 | 주력 심사 항목 | 주력 ES 기능 |
|---|---------|-------------|------------|
| 감성/소셜 | Taste Translator | 데모 30% + 소셜 10% | 시맨틱 검색, RRF, ELSER |
| 기술/분석 | Unseen Gap Detector | 기술 실행력 30% | ES|QL, 어그리게이션, 벡터 밀도 |
| 발견/혁신 | Serendipity Engine | 영향력·혁신성 30% | 하이브리드 검색, RRF, 2-hop |

## 보완된 관점

숙의를 통해 발견된 추가 고려사항:

1. **데이터 확보가 최대 변수** — 14일 1인 개발에서 데이터 정제가 일정의 대부분을 잡아먹을 수 있음. 라이선스가 명확하고 정제된 오픈 데이터셋 우선
2. **"단순 RAG"로 보이면 안 됨** — 검색→집계/ES|QL→다음 도구 호출→검증→요약의 반복 루프가 화면에 드러나야 Agent Builder 해커톤의 필수조건 충족
3. **3개 후보는 실제 제출 1개를 위한 전략** — 가장 먼저 데이터 확보 + 프로토타입이 되는 것을 최종 1개로 확정
